# Make the RunnerPool CR the real single source for Engine Version

The RunnerPool custom resource is the single declarative source for an Engine
Version and its per-User cap (ADR-0006). The live operator already honours this
for the cap and the KEDA query; the residual leaks are the **Runner image tag**
inside a CR and the **Compose path**, which re-states every literal independently.

Status: proposed

## Context

A correction to this candidate's first framing (the live KEDA leak it described is
mostly not real):

- **The live KEDA query already derives from the CR.** The operator interpolates
  `engine_version` from `pool.Spec.EngineVersion` and the cap from
  `caps(pool.Spec.DefaultUserConcurrency, 50)` into the generated `ScaledObject`
  query — `internal/controller/runnerpool_controller.go:161,165-169`. The
  standalone `config/keda/scaledobject.yaml` (`< 50`, `'24.1.0'`) is explicitly a
  **documentation sample** (header `:1-9`), not the object that runs.
- **`24.1.0` appears across ~8 live-manifest sites** in 5 files
  (`config/samples/runnerpool.yaml:7,8`, `config/keda/scaledobject.yaml:29` [the
  sample], `deploy/docker-compose.yml:61,105`,
  `deploy/k8s-local/runnerpool-demo.yaml:10,11`, `deploy/k8s-local/secret.yaml:21`)
  — and in 30+ files overall once build tags, Dockerfile ARGs, SDK and seed data
  are counted, which are legitimately separate and out of scope here.
- **The cap `50` is two different kinds** that must not be conflated: live-manifest
  literals (`runnerpool.yaml:14`, the KEDA sample, `docker-compose.yml:108`,
  `runnerpool-demo.yaml:17`) vs. code/schema fallback defaults
  (`config.go:44`, `runnerpool_controller.go:108,161`, `config.py:65`,
  `runnerpool_types.go:44`). Only the former are duplicated *configuration*.

So the genuine residuals are narrow: **(a)** the Runner image tag is the one value
inside a CR that does not derive from `spec.EngineVersion` — `image:
ghcr.io/synergyplus/energyplus-runner:24.1.0` is typed verbatim
(`runnerpool_controller.go:123` consumes `pool.Spec.Image` as-is); and **(b)** the
Compose path has no operator, so it re-states `SP_ENGINE_VERSION` /
`SP_ALLOWED_ENGINE_VERSIONS` by hand.

## Decision

- **Image tag:** make `spec.Image` optional and have the operator default it from
  `spec.EngineVersion` (a convention or defaulting webhook), so Engine Version is
  stated once per CR.
- **Compose & secret envs:** introduce one tracked per-version value file (a "pool
  catalog") as the sole authored home for the set of `engineVersion` entries;
  render it into `runnerpool.yaml`, `runnerpool-demo.yaml`, `secret.yaml`'s
  `SP_ALLOWED_ENGINE_VERSIONS`, and the Compose envs. **Commit the rendered
  output** (generated by `make generate` / CI) so the `docker compose up` path
  needs no pre-render step, and have CI fail if the checked-in output drifts from
  the catalog.

This candidate does **not** depend on ADR-0011: the cap is already single-sourced
in the live operator path, so there is no cap-duplication for ADR-0011 to "absorb"
here. The two ADRs are independent.

## Considered options

- **Operator-defaulted image tag + a committed-render catalog (preferred)** —
  removes the two real residuals; honours ADR-0006 without breaking the
  zero-tooling Compose path.
- **A single Kustomize/Helm value** feeding all manifests — addresses the Compose
  sprawl but not the in-CR image-tag derivation, and adds a render step to the
  local path.
- **Status quo** — version bumps stay a small checklist; acceptable because the
  set of versions changes rarely. This is the weakest candidate.

## Consequences

- **Locality:** Engine Version is declared once per CR; the version set is
  declared once in the catalog.
- **Leverage:** a version bump becomes a one-line spec/catalog change with CI
  guarding drift.
- **Lowest urgency of the five.** The dangerous duplication (the eligibility
  *predicate*) is ADR-0011's, not this one; the residuals here are deploy-manifest
  ergonomics. Sequence it last.
